# CSS AST 等価性チェックの説明

## この文書の目的

この文書は、Emotion で書かれた CSS と、SCSS で書かれた CSS が「同じ見た目になる」と判断できる理由を、QA、デザイナー、PM など非エンジニアの方にも分かるように説明するものです。

結論から言うと、このツールは単なる文字の一致ではなく、ブラウザが解釈する CSS の構造に近い形まで分解して比較します。そのため、改行や空白の違いに惑わされず、見た目に影響する重要な部分が同じかどうかを確認できます。

## なぜこの確認が必要なのか

Emotion から SCSS へ移行する場合、見た目が変わらないことがとても重要です。

たとえば次のような不安があります。

- ボタンの色が変わらないか
- hover 時の見た目が壊れないか
- 余白や角丸が変わらないか
- レイアウトが崩れないか
- レスポンシブ指定が抜け落ちないか

このツールは、Emotion 側と SCSS 側を比較して、CSS として同じ意味を持っているかを確認します。

## AST とは何か

AST は Abstract Syntax Tree の略で、日本語では抽象構文木と呼ばれます。

難しく聞こえますが、ここでは「CSS を比較しやすい形に整理したもの」と考えてください。

たとえば、次の CSS があります。

```css
.Button {
  color: red;
}
```

人間にはこのままでも読めますが、プログラムにとってはただの文字列です。AST にすると、次のような意味のまとまりとして扱えます。

```json
{
  "selector": ".Button",
  "declarations": [
    {
      "property": "color",
      "value": "red"
    }
  ]
}
```

つまり AST は、CSS を次のような部品に分けたものです。

- どの要素に当てるか: selector
- どの CSS プロパティか: property
- どの値か: value
- media query などの条件: at-rule
- 宣言の順番

## なぜ文字列比較では不十分なのか

CSS は、書き方が少し違っても同じ意味になることがあります。

たとえば次の2つは、見た目としては同じです。

```css
.Button {
  color: red;
}
```

```css
.Button{color:red;}
```

文字列としては違います。

- 改行が違う
- 空白が違う
- セミコロンの有無が違う

しかし、CSS としての意味は同じです。文字列比較だけだと、このような安全な違いまで「差分あり」と判定してしまいます。

そこでこのツールでは、一度 CSS AST に変換してから比較します。

## このツールが比較しているもの

このツールは、次の流れで確認します。

1. Emotion の CSS を取り出す
2. 同じファイル名、同じディレクトリにある SCSS を探す
3. Emotion 側と SCSS 側を CSS として解釈できる形にする
4. それぞれを AST に変換する
5. AST 同士を比較する

比較対象は主に次の内容です。

- selector が同じか
- at-rule が同じか
- CSS property が同じか
- CSS value が同じか
- 同じ selector 内の宣言順が同じか

これらは、画面表示に直接影響しやすい要素です。

## 等価とは何を意味するのか

このツールで「等価」と判定された場合、次のことが言えます。

- Emotion 側と SCSS 側で、同じ selector が存在する
- 同じ CSS property が存在する
- property に対する value が同じ
- hover や media query などの構造が同じ
- CSS の宣言順が維持されている

つまり、少なくともこのツールが見ている範囲では、Emotion から SCSS へ移しても CSS の意味は変わっていないと判断できます。

QA 観点では、これは「見た目が変わる可能性のある CSS 差分が検出されていない」状態です。

デザイナー観点では、これは「色、余白、状態変化、条件付きスタイルなどの指定が SCSS 側にも同じ形で残っている」状態です。

PM 観点では、これは「移行による CSS 崩れのリスクを機械的に確認できている」状態です。

## 等価性チェックで安心できること

このチェックに通ることで、次のような不安を減らせます。

- 色指定の抜け漏れ
- hover や focus など状態スタイルの抜け漏れ
- padding、margin、font-size などの値の変更
- media query の抜け漏れ
- selector の変更
- 宣言順の変更による上書き結果の変化

特に CSS は、後に書いた宣言が前の宣言を上書きすることがあります。そのため、このツールでは同じ selector 内の宣言順も比較対象にしています。

## 保証できないこと

このチェックは強力ですが、すべてを保証するものではありません。

たとえば次の内容は、別途確認が必要です。

- JavaScript の条件分岐によって実行時に変わるスタイル
- ブラウザごとの差異
- HTML 構造の変更による見た目の変化
- class name の付け方や適用箇所の変更
- CSS 以外の要因によるレイアウト崩れ
- 画像、フォント、外部ライブラリの読み込み状態

つまり、このツールは「CSS 定義そのものが同じ意味になっているか」を確認するものです。実際の画面確認やビジュアルリグレッションテストの代わりではなく、それらを補助する安全確認です。

## デバッグ用 AST ファイルの見方

デバッグモードを使うと、比較に使った AST をファイルとして出力できます。

```powershell
go run .\cmd\emotion-to-scss check test-css\basic-ok --debug-ast
```

出力例:

```text
test.emotion.ast.json
test.scss.ast.json
```

ファイル名の意味は次のとおりです。

- `*.emotion.ast.json`: Emotion 側から作った AST
- `*.scss.ast.json`: SCSS 側から作った AST

この2つを見比べると、どの selector、property、value が比較対象になったかを確認できます。

## QA での使い方

QA では、次のように使えます。

1. Emotion から SCSS へ移行したファイルに対してチェックを実行する
2. `OK` になっていることを確認する
3. 必要に応じて `--debug-ast` で AST を出力する
4. 差分がある場合は、どの selector や property が違うかを開発者に確認する

`OK` であれば、CSS 定義の移行に関しては機械的な一致確認が取れている状態です。

## デザイナー向けの見方

デザイナーは、AST の細かい構造をすべて読む必要はありません。

見るべきポイントは次のようなものです。

- `Selector` が期待する class になっているか
- `Property` に `color`、`padding`、`font-size` など期待する指定があるか
- `Value` がデザイン指定と一致しているか
- hover や media query の指定が残っているか

たとえば、ボタン色が変わっていないかを見る場合は、`background-color` や `color` の値を確認します。

## PM 向けの判断材料

PM は、このチェック結果を移行リスクの判断材料として使えます。

`OK` の場合:

- CSS 定義の意味が一致している
- 移行による見た目崩れの可能性が低い
- 追加の画面確認に進める状態

`FAIL` の場合:

- CSS 定義に意味の差がある
- 見た目が変わる可能性がある
- 差分の確認と修正が必要

このように、チェック結果をリリース判断や QA 優先度の整理に使えます。

## まとめ

AST 等価性チェックは、Emotion と SCSS の CSS 定義が同じ意味を持っているかを機械的に確認する仕組みです。

文字列の見た目ではなく、CSS の意味に近い構造で比較するため、不要な差分に惑わされにくく、移行による CSS 崩れのリスクを減らせます。

ただし、これは画面確認を完全に置き換えるものではありません。CSS 定義の安全確認として使い、そのうえで実際の画面や主要導線を確認することで、より安心して移行を進められます。

## エンジニアへのメッセージ

エンジニア目線でも、このツールの等価性チェックだけで「UI 崩れは絶対に起こりえない」とまでは言い切れません。

このツールが証明しているのは、主に次の2つが CSS として同じ意味を持つことです。

- Emotion から抽出した CSS
- SCSS をコンパイルした CSS

具体的には、selector、at-rule、property、value、宣言順が一致していることを確認します。そのため、CSS 定義そのものが原因で見た目が変わる可能性はかなり低くなります。

一方で、UI の見た目は CSS 定義だけでは決まりません。次の条件も同じである必要があります。

- DOM 構造が同じ
- className の付与先が同じ
- CSS の読み込み順や挿入順が同じ
- 他の CSS との cascade や specificity の関係が同じ
- CSS custom property の値が同じ
- JavaScript の条件分岐で出る動的スタイルが同じ
- Emotion の runtime 挙動と SCSS の適用タイミングが同じ
- ブラウザや実行環境の差がない

特に Emotion から SCSS へ移す場合、CSS の挿入順が変わる可能性があります。CSS は selector、property、value が同じでも、他の CSS との前後関係が変わると最終的な見た目が変わることがあります。

したがって、より正確には次のように捉えるのが安全です。

> 比較対象の CSS について、正規化後 AST が等価であり、かつ DOM、className、読み込み順、実行時値、周辺 CSS が変わらないなら、その CSS 差分による UI 崩れは起きないと考えてよい。

このツールは、UI 崩れが絶対に起きないことの完全証明ではありません。CSS 移行によって見た目が変わる主要原因を、かなり強く潰すための検証です。

実務上は、「CSS の意味は一致している」と機械的に言えることに価値があります。これにより、残る確認ポイントを DOM、className、読み込み順、画面テストなどに絞ることができ、レビューや QA の負担を減らせます。
